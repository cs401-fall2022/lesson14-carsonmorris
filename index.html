Learn Typescript: For Javascript Programmers

Introduction

This page will cover the basics of Typescript and its corresponding relationship with Javascript. Typescript and Javascript have an unusual relationship, where Javascript code is already Typescript code. For each section below, we will cover some of the basics of Typescript.

Types in Typescript

In Typescript, you can indicate a variable's type based on what you assign it to. Instead of declaring that myString is a string, you can just set it equal to a string. See below:

let myString = "Hello cruel and evil world.";

This offers a type system that does not need to be explicitly defined. Typescript knows that myString is a string based on what it is assigned to.

Defining Types

Sometimes there is difficulty with inferring types automatically. To correct this, TypeScript supports an extension of JavaScript which allows you to tell TypeScript what the types should be. You can specifically describe an object using an interface, such as what we see below:

interface user {

name: string;

id: number;

}

You can also create objects with inferred types such as what we see below:

const user = {

name: "Carson Morris";

id: 123;

}

You can then declare that a JavaScript object conforms to your new interface by using syntax like : TypeName after a declaration:

const user : User = {

name: "Carson Morris";

id: 123;

}

If you provide an object that doesn’t match the interface, TypeScript will warn you:

interface user {

name: string;

id: number;

}

const user : User = {

incorrectName: "Carson Morris";

Type '{ incorrectName: string; id: number; }' is not assignable to type 'User'. Object literal may only specify known properties, and 'incorrectName' does not exist in type 'User'.

id: 123;

}

Composing Types

With Typescript you can create your own types. For example, you could create the Boolean type by utilizing "Unions" by doing the following:

type myBoolean = True | False;

You can also use this to set what literals are allowed to be. Some examples would be:

type doorState = "locked" | "unlocked";

type bestStarWarsProjects = "Revenge of the Sith" | "The Mandalorian" | "Rogue One";

type evenNumbersUnderTen = 2 | 4 | 6 | 8;

typeof

Typescript also makes use the typeof, which can be used to ensure type usage. Below is a table of potential uses for this method.

Type Predicate
String typeof s === "string"
number typeof n === "number"
bolean typeof b === "boolean"
undefined typeof undefined === "undefined"
function typeof f === "function"
array Array.isArray(a)

Example of Using typeof:

function funcName (obj: string | string[]) {

if(typeof obj == "string") {

//do something

}

}

Generics

You can also use generics in Typescript. A common example is with Arrays. An array without generics can contain anything, but specifying generics describes what the array can hold.

type StringArray = Array<string>;

type NumberArray = Array<number>;

type ObjectWithNameArray = Array<{ name: string }>;

JS Projects Utilizing TypeScript

The type system in TypeScript has different levels of strictness when working with a codebase:

  • A type-system based only on inference with JavaScript code
  • Incremental typing in JavaScript via JSDoc
  • Using // @ts-check in a JavaScript file
  • TypeScript code
  • TypeScript with strict enabled

Each step represents a move towards a safer type-system, but not every project needs that level of verification.

Providing Type Hints in JS via JSDoc

In a .js file, types can often be inferred. When types can’t be inferred, they can be specified using JSDoc syntax.

JSDoc annotations come before a declaration will be used to set the type of that declaration. For example:

/**@type {number}*/

var x;

x = 0; //Ok

x = false; //Ok!?

The last line of the previous code sample would raise an error in TypeScript, but it doesn’t by default in a JS project. To enable errors in your JavaScript files add: // @ts-check to the first line in your .js files to have TypeScript raise it as an error.

// @ts-check

/**@type {number}*/

var x;

x = 0; //Ok

x = false; //Not Ok

Rethinking the class

C# and Java are classic examples of Object Oriented Programming (OOP) languages. In these languages, the class is the basic unit of code organization. Forcing all functionality and data to be held in classes can be a good domain model for some problems, but not every domain needs to be represented this way.

In JavaScript, functions can live anywhere, and data can be passed around freely. This flexibility is extremely powerful. “Free” functions (those not associated with a class) working over data without an implied OOP hierarchy tends to be the preferred model for writing programs in JavaScript.

That said, you can still use classes if you like! Some problems are well-suited to being solved by a traditional OOP hierarchy, and TypeScript’s support for JavaScript classes will make these models even more powerful. TypeScript supports many common patterns such as implementing interfaces, inheritance, and static methods.
